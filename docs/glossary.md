# Glossary

Terms that recur in ContentOps docs, CLI output, and runbooks. Linked
from [CLAUDE.md](../CLAUDE.md) and
[OPERATOR_GUIDE.md](OPERATOR_GUIDE.md); add here on first non-obvious
use anywhere else.

## A

### ARM (Azure Resource Manager)
Azure's control-plane REST API (`management.azure.com`). Sentinel analytic / hunting / watchlist / parser / data-connector resources are created and updated via ARM PUTs under `Microsoft.SecurityInsights` + `Microsoft.OperationalInsights` (API version `2025-07-01-preview`). Defender custom detections use Microsoft Graph instead (see *Graph*). See [`architecture.md`](reference/architecture.md).

### Asset kind
One of the six supported detection-content categories
(see [`docs/reference/asset-coverage.md`](reference/asset-coverage.md)):
`sentinel_analytic`, `sentinel_hunting`, `sentinel_watchlist`,
`sentinel_parser`, `sentinel_data_connector`,
`defender_custom_detection`. Each kind has one handler in
`contentops/handlers/` implementing the
`validate → plan → apply → delete` protocol.

### Audit chain
Append-only JSONL files under `audit/<date>.jsonl`. Each record
carries `prev_hash` (SHA-256 of the previous record) plus
`record_hash` (its own hash), forming a forward chain. The chain is
**monotonic**: within a batch the writer bumps each record's
timestamp to `max(record.timestamp, prev_timestamp + 1µs)` so the
chain stays strictly forward-ordered even across rapid batches.
Verified by `contentops audit verify` and the weekly
`audit-verify.yml` workflow. See
[`docs/reference/audit-trail.md`](reference/audit-trail.md).

### Auto-disabled rule
A Sentinel analytic rule the platform has switched off itself
(consecutive query failures, table schema break, ingest gap, etc.).
Distinct from a *silent rule* (one that simply produced no alerts).
Surfaced by `contentops auto-disabled-rules` which unions
`SentinelHealth.Status in ("Disabled", "Failure")` with
`LAQueryLogs` recent-failure rows. NVISO Part 7 borrowing.
**Prerequisite:** the `SentinelHealth` diagnostic data collection
must be enabled on the workspace — `contentops doctor --auth`
includes a probe that warns when zero rows are returned.

## C

### Cohort
An optional free-text grouping tag on an envelope (`metadata.cohort`). Lets an operator act on a named set of detections at once — e.g. `contentops disable --cohort NAME`, `contentops enable --cohort NAME`, `contentops portfolio --cohort C`.

### Conformance
A read-only, layered health check of the install. Levels L1–L7 cover
local install, tenant config, OIDC, Graph permissions, Azure RBAC,
functional reach, and GitHub repo settings. Run via
`contentops conformance`. Used pre-flight on a new clone and
weekly via `conformance.yml`. See
[`docs/operations/deployment-conformance.md`](operations/deployment-conformance.md).

## D

### D3FEND
MITRE's **defensive** technique taxonomy (https://d3fend.mitre.org/),
companion to ATT&CK. Where an ATT&CK technique names attacker
behaviour (`T1059` = Command and Scripting Interpreter), a D3FEND
technique names a defensive countermeasure (`D3-PSA` = Process Spawn
Analysis). ContentOps tracks D3FEND via the optional
`metadata.defensiveTechniques: [D3-XXX]` field on every envelope;
`contentops coverage --d3fend` renders a per-tactic coverage report
from the bundled curated list at
`contentops/coverage/data/d3fend_techniques.json`.

### Detection documentation (per-detection)
The `docs/detections/<asset>/<id>.md` markdown pages generated by
`contentops detection-docs regenerate`. One page per envelope; pulls
metadata + KQL/payload preview into a SOC-analyst-readable shape.
Drift-gated by the pytest suite (`test_committed_detection_docs_are_in_sync`)
so a forgotten regen blocks PRs. NVISO Part 4 borrowing.

### Drift PR
A bot-generated pull request from `drift.yml` when the in-tenant
state of a detection differs from what `detections/<kind>/*.yml`
says it should be. The PR carries a YAML diff so a reviewer can
either accept the tenant change (merge) or restore the repo state
(close + re-apply). Distinct from a content PR authored by a human.

### Drift suppressions
Operator-curated allowlist at `detections/drift_suppressions.yml`
naming `(asset, id)` pairs that should NOT show up in drift reports
(e.g. an analyst tuned a threshold in the portal pending a data
review). Each entry carries a `reason` and an `expires` date;
expired entries re-surface tagged `[suppression-expired]`. The
NVISO Part 8 `contentops tuning preview` command posts a PR comment
with the 30-day blast-radius (incidents + alerts) for each new
suppression entry so reviewers see what they're about to silence.

### Drift-resolve strategies
The `--strategy` choices for `contentops drift-resolve <id>`: `git` (local YAML wins; no tenant change), `remote` (write the tenant's current version back into local YAML), and `merge` (3-way editor reconcile — currently deferred; raises a not-implemented error). The per-rule counterpart to accepting or rejecting a whole *Drift PR*.

### Draft / experimental
Not a status value (see *Status*). Colloquially, an envelope still
being authored — `status: experimental`. Lint placeholders
(`TODO (METAxxx): ...`) are accepted; META008 escalates them once
status moves past `experimental`.

## E

### Envelope
The canonical YAML file for one piece of detection content. Has
`id`, `version`, `asset`, `status`, optional `metadata` block,
and a `payload` block matching the asset kind's deployable shape.
See `contentops/core/envelope.py` and
[`docs/reference/envelope-schema.md`](reference/envelope-schema.md).

### Env-status gate
The filter in `contentops/core/env_status.py` that drops envelopes
whose `status:` value isn't allowed in the active workspace. The
mapping by role:
- `role: prod` → accepts `production`, `deprecated`
- `role: integration | staging | stage` → accepts `test`,
  `production`, `deprecated` (shared lower env)
- `role: test` → accepts `test`, `deprecated` only (dedicated test
  workspace; production envelopes do NOT spill in — closes G21)
- `role: dev | development | sandbox | local` → accepts everything
Plus a Defender-specific carve-out: Defender custom detection
envelopes only deploy to `role: prod` (Defender XDR is tenant-wide,
no per-workspace routing).

### ETag
An HTTP concurrency token that Sentinel ARM resources expose. Write-capable Sentinel handlers read the remote first, capture its ETag, and PUT with `If-Match`; a `412 Precondition Failed` surfaces as a "rerun `contentops plan` and resolve drift" message rather than a stack trace. The Defender Graph beta API has no ETag, so Defender writes rely on post-apply hash verification only.

## G

### Graph (Microsoft Graph)
Microsoft's unified API (`graph.microsoft.com`). ContentOps uses the Graph **Security beta** endpoint (`security/rules` / `detectionRules`) to deploy Defender XDR custom detections, and Graph `alerts_v2` for the alert pipeline. The Sentinel side uses *ARM*, not Graph.

## H

### Handler
The Python class responsible for one asset kind in
`contentops/handlers/`. Implements `validate()`, `plan()`,
`apply()`, `delete()` against the relevant Microsoft API.
Registered in `contentops/core/registry.py`.

## I

### Integration vs Production
Two `role` values a Sentinel workspace can carry in `tenant.yml`.
`integration` runs as a PR-time smoke test for detection content;
`production` runs on push to `main`. A tenant can have many
workspaces of each role. See
[`docs/operations/multi-workspace.md`](operations/multi-workspace.md).

## K

### KQL (Kusto Query Language)
The query language used by Azure Sentinel / Log Analytics and Defender Advanced Hunting (https://learn.microsoft.com/kusto/query/). A detection's logic lives in `payload.query` (Sentinel) or `payload.queryCondition.queryText` (Defender) as KQL. The `KQL0xx` / `KQL101` lint rules statically check it; the full lint-rule list is in [`generated-catalog.md`](reference/generated-catalog.md) under "Lint rules".

## L

### L1 – L7 (conformance layers)
The seven conformance check tiers. L1 is the lightest (local
install); L7 is the heaviest (GitHub repo settings via `gh`).
See *Conformance*.

### Lifecycle stage
Optional `lifecycleStage` field on the envelope —
`concept | research | engineering | delivery | optimization |
feedback`. Orthogonal to `status` (which drives apply behaviour);
this drives dashboards and team-lead planning. Never gates CI.
See `contentops/core/lifecycle_stage.py`.

### localCustomization
A top-level boolean flag on an envelope. When `true`, `contentops apply` **skips** that asset unless `--force-overwrite` is passed — so an analyst's in-portal hand-tuning isn't flattened by a bulk apply. Set / cleared via `contentops lock` / `contentops unlock` (which open a PR via `lock-unlock.yml`). See *Lock*.

### Lock
A boolean `localCustomization: true` flag on an envelope that
tells `contentops apply` to skip the asset unless
`--force-overwrite` is set. Set / cleared via `contentops lock` /
`contentops unlock`, which open a PR through `lock-unlock.yml`.

## M

### MITRE ATT&CK
The framework of adversary tactics and techniques (https://attack.mitre.org/). Each technique has an ID like `T1098` (with sub-techniques like `T1098.003`). Detections map to tactics/techniques via `metadata`/`payload` fields; `contentops coverage` renders an ATT&CK heatmap and `contentops navigator` a Navigator layer. See also *D3FEND* (the defensive companion taxonomy).

### MITRE mapping
The `tactics` + `techniques` lists on the payload (Sentinel
kinds) or `mitreTechniques` on the alert template (Defender).
PAYLOAD003 lint warns when either is empty.

### MITRE Navigator layer
The JSON file format consumed by the hosted MITRE ATT&CK Navigator
UI at https://mitre-attack.github.io/attack-navigator/. ContentOps
generates one via `contentops navigator`, aggregating three coverage
axes (repo envelopes / deployed rules / live `SecurityAlert`
firings) into a single layer where each technique tile carries a
score = unique-rule-name count across selected axes. The hosted UI
renders the JSON for free; the project intentionally does NOT ship
an SVG export to keep the runtime dependency surface lean.

## P

### Payload
The asset-kind-specific block inside an envelope —
the bytes that get sent to ARM or Microsoft Graph. Distinct from
`metadata` (authoring/triage info managed at PR time).

### Provider
A thin wrapper around one backend HTTP API that handlers compose. The Sentinel provider is `contentops/providers/sentinel_arm.py` (`SentinelArmProvider`, ARM REST); the Defender path is `contentops/defender/client.py` + `contentops/defender/deploy.py` (Graph beta). Handlers hold the per-asset-kind logic; providers hold the transport.

### Public mirror
The `SecM8/ContentOps` repo on GitHub, rebuilt nightly from
this private operator repo via `public-sync.yml` and
`.github/sync-allowlist.txt`. One-way; never receives PRs back.

## R

### Role (workspace role)
The `role:` tag on each Sentinel workspace in `config/tenant.yml` — `prod` | `integration` | `test` | `dev` (plus documented synonyms). It selects which workspace(s) a command targets (`--role` / `--workspace`) and which envelope `status` values may deploy there (see *Env-status gate*). Defender XDR is tenant-wide and only deploys to `role: prod`.

## S

### scaffoldStrict
The `policy.scaffoldStrict` knob in `config/tenant.yml`. Controls
whether the META002–META005 authoring-metadata lint rules
(description / attackDescription / references / falsePositives)
fire as **warnings** (lenient) or **errors** (strict, CI-blocking).
**Default since PR #241 is False (lenient)** so adopters with a
fresh tenant.yml aren't blocked by the G24 authoring backlog out of
the box. Operators with a drained backlog set `scaffoldStrict: true`
to upgrade to CI-blocking. See `contentops/config.py:is_scaffold_strict()`.

### Silent rule
A deployed analytic rule that has fired zero alerts over a recent
window. Reported by `contentops silent-rules` and the
`silent-rules.yml` workflow. Not automatically retired —
operator decides.

### Snippet
A reusable KQL fragment under `overrides/`. Applied by
`contentops/snippets/apply.py` during apply, so the same FP
allowlist or noise filter can be referenced from multiple rules.

### State branch
Refs `refs/heads/state/<env>` (e.g. `state/prod`,
`state/integration`). Each carries one file — `state/state.json`
— summarising the last successful deploy. Pulled at the start of
each `apply` (so the runner knows what's already deployed) and
pushed at the end (so the next runner starts current).

### Status
The string on `envelope.status`. One of `experimental | test |
production | deprecated`. Drives lint severity for some rules
(META001 freshness threshold, META008 placeholder escalation)
and is independent of `lifecycleStage`.

## T

### Tenant config
`config/tenant.yml`. Gitignored. Carries the Sentinel workspace
list (subscription IDs, RGs, workspace names) plus the App
Registration client ID for OIDC. Materialised in CI from the
`TENANT_CONFIG_YAML` secret. See
[`docs/operations/tenant-config-modes.md`](operations/tenant-config-modes.md).

### Test workspace (dedicated `role: test`)
A Sentinel workspace tagged `role: test` in `tenant.yml`.
Distinct from `role: integration`: a dedicated test workspace
accepts ONLY envelopes with `status: test` (plus `deprecated`),
keeping test workloads isolated from live production content. Closes
G21. Operators who prefer the shared-lower-env pattern (test +
production both deploy together) continue to use `role: integration`.

### Tuning preview
The PR-time blast-radius comment posted by
`.github/workflows/tuning-impact-preview.yml` when a contributor
adds a `detections/drift_suppressions.yml` entry. Runs
`contentops tuning preview`, which executes a 30-day retrospective
KQL against the workspace and reports "this suppression would have
silenced N incidents and M alerts". Fork-PR safe (the workflow
detects fork PRs and renders the comment with `-` placeholders for
the count columns). NVISO Part 8 borrowing.
